## 3. 验收用例

### TC-01: 后端 API 可用性

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 打开 Swagger UI `http://localhost:3000/api/docs` | 显示 API 文档列表 |
| 2 | 展开 `Auth` 模块，点击 `POST /api/auth/login` | 显示接口参数 |
| 3 | 点击 "Try it out"，输入 `{ "username": "admin", "password": "admin123" }` | 返回 200，包含 accessToken 和 refreshToken |
| 4 | 点击 `GET /api/auth/me` | 显示接口信息 |
| 5 | 在 Authorization 输入 `Bearer <accessToken>`，点击 Execute | 返回当前用户信息 |

**通过标准：** 登录接口返回 Token，me 接口返回用户数据

---

### TC-02: 前端登录页面

| 3 | 点击登录按钮 | 显示 "登录成功"，跳转首页 |
| 4 | 检查浏览器 localStorage | 存在 `token` 和 `refreshToken` |

**通过标准：** 登录成功，Token 正确存储，跳转正常

### TC-08: Token 刷新机制

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 正常登录 | 获取 Token |
| 2 | 等待 Access Token 过期（2小时）或手动修改 localStorage 的 token 为无效值 | - |
| 3 | 触发任意需要认证的 API 请求（如点击菜单获取数据）| 后端返回 401 |
| 4 | 前端自动调用 `/api/auth/refresh` | 使用 Refresh Token 获取新 Access Token |
| 5 | 原请求自动重试 | 请求成功，数据正常返回 |
| 6 | 检查 localStorage | Token 已更新为新值 |

**通过标准：** Token 过期后自动刷新，用户无感知
